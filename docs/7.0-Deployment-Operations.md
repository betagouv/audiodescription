# Déploiement et opérations

Ce document couvre l'architecture de déploiement, les procédures opérationnelles et les tâches de maintenance du Portail de l'audiodescription en production. 
Il explique la configuration Docker, la planification des imports automatisés, la gestion des logs et les commandes du quotidien.

Pour les informations sur la configuration locale de développement, voir [Démarrage](3-Demarrage.md). 
Pour les détails sur la logique du pipeline d'import et les intégrations partenaires, voir [Pipeline d'import des données](6.1.0-Pipeline-Import.md).

## Vue d'ensemble de l'environnement de production

Le portail audiodescription s'exécute comme une application Docker multi-conteneurs orchestrée par Docker Compose.

La production utilise **Traefik** pour la terminaison HTTPS et le routage par hôte. 

Les services `patrimony` et `drupal` exécutent des applications PHP distinctes avec **Caddy** comme serveur web. 
Les deux services se connectent à une instance PostgreSQL partagée avec des bases séparées. 
Elasticsearch tourne comme service dédié pour la recherche.

## Configuration Docker Compose

### Service Patrimony

Le service `patrimony` exécute l'application Symfony backend responsable des intégrations d'API partenaires et de la canonicalisation des données.

| Configuration | Valeur |
|---|---|
| **Dockerfile** | `docker/patrimony/prod/php/Dockerfile` |
| **Répertoire de travail** | `/app/public/patrimony` |
| **Port exposé** | 8081 |
| **Hôte Traefik** | `patrimony.audiodescription.beta.gouv.fr` |
| **Politique de redémarrage** | `always` |

Volumes montés : code applicatif `.:/app/public`, configuration Caddy et configuration PHP runtime.

La configuration Traefik inclut un middleware `strip-www` pour canonicaliser les URLs vers la version sans www.

### Service Drupal

Le service `drupal` héberge le CMS et le site public.

| Configuration | Valeur |
|---|---|
| **Dockerfile** | `docker/drupal/prod/php/Dockerfile` |
| **Répertoire de travail** | `/app/public/drupal` |
| **Port exposé** | 8080 |
| **Hôtes Traefik** | `audiodescription.beta.gouv.fr`, `www.audiodescription.beta.gouv.fr` |
| **Politique de redémarrage** | `always` |
| **Dépendance** | service `db` |

Volumes montés : code applicatif `.:/app/public`, configuration Caddy et configuration PHP runtime.

La configuration Traefik inclut un middleware `strip-www` pour canonicaliser les URLs vers la version sans www.

### Service Base de données

PostgreSQL 17 fournit le stockage persistant pour les deux applications.

| Configuration | Valeur |
|---|---|
| **Image** | `postgres:17-bookworm` |
| **Volume** | `db-data:/var/lib/postgresql/data` |
| **Script d'init** | `./docker/postgres/init-patrimony-db.sh` |
| **Variables d'env.** | `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD` |

Le script d'init crée deux bases : `patrimony_db` et `drupal_db`, permettant aux deux applications de partager une seule instance PostgreSQL avec isolation des données.

### Service Elasticsearch

Elasticsearch 8.14.3 alimente la fonctionnalité de recherche de films.

| Configuration | Valeur |
|---|---|
| **Image** | `docker.elastic.co/elasticsearch/elasticsearch:8.14.3` |
| **Type de découverte** | `single-node` |
| **Sécurité** | Désactivée (`xpack.security.enabled=false`) |
| **Taille du tas** | 2 Go min/max (`-Xms2048m -Xmx2048m`) |
| **Volume** | `elastic-data:/usr/share/elasticsearch/data` |

### Architecture réseau et volumes

Le réseau `web` est externe et préexistant, connecté au reverse proxy Traefik. 
Le réseau `ad-network` est interne, isolant le trafic base de données et Elasticsearch de l'accès externe. 
Seuls `patrimony` et `drupal` exposent des ports à Traefik.

| Volume | Rôle | Monté dans |
|---|---|---|
| `db-data` | Fichiers de données PostgreSQL | `/var/lib/postgresql/data` |
| `elastic-data` | Index Elasticsearch | `/usr/share/elasticsearch/data` |

Le code applicatif est monté directement depuis le système de fichiers hôte via `.:/app/public`, permettant les déploiements sans reconstruire les images.

## Cron et automatisation

### Planification de l'import quotidien

Le système exécute des imports automatisés quotidiennement à **01h05** via le cron système.

```
5 1 * * * cd /opt/audiodescription && make cron-import >> /opt/audiodescription/logs/cron-import.log
```

La cible `cron-import` exécute les imports avec tolérance aux erreurs via le flag `-T` (mode non-TTY) et `|| true` pour qu'une erreur n'arrête pas l'exécution des imports suivants.
Chaque commande d'import inclut `--create-movies=true` pour activer la canonicalisation et la création automatique de films. L'import CSV CNC est **exclu** du cron car les données de référence CNC sont mises à jour moins fréquemment.


### Planification de l'envoi de la newsletter hebdomadaire
Le système envoie une newsletter automatisée tous les lundis à **12h30** via le cron système.

```
30 12 * * 1 cd /opt/audiodescription && make pt-send-newsletter > /var/log/audiodescription/cron-newsletter.log 2>&1
```

### Rotation des logs

Les logs de production sont gérés par logrotate avec une rétention de 15 jours.

```
/var/log/audiodescription/cron-import.log {
    daily
    rotate 15
    compress
    delaycompress
    missingok
    notifempty
    create 644 deployer deployer
}
```

| Paramètre | Valeur | Rôle |
|---|---|---|
| Fréquence | `daily` | Rotation une fois par jour |
| Rétention | `rotate 15` | Conserver 15 fichiers archivés |
| Compression | `compress` | Gzip les anciens logs |
| Délai compression | `delaycompress` | Compresser à la rotation suivante |
| Fichier absent | `missingok` | Ne pas faire d'erreur si absent |
| Vide | `notifempty` | Ignorer la rotation si vide |
| Propriétaire | `deployer:deployer` | Owner/groupe du fichier |
| Permissions | `644` | Lecture/écriture owner, lecture autres |

## Procédures de déploiement

### Déploiement initial en production

**1. Démarrer les services :**
Exécute `docker compose -f compose.prod.yml up -d --build --remove-orphans -t 0`.

**2. Installer Patrimony (`prd-install-patrimony`) :**
- `composer install` — installation des dépendances PHP
- `doctrine:database:create --if-not-exists` — création de la base PostgreSQL
- `doctrine:migrations:migrate --no-interaction` — exécution des migrations
- `c:c` — nettoyage du cache Symfony
- `ad:import:cnc-public` — import des données de référence CNC
- Import depuis tous les partenaires avec `--create-movies=true`

**3. Installer Drupal (`prd-install-drupal`) :**
- `composer install` — installation des dépendances PHP
- `drush si -y` — installation fraîche de Drupal
- `drush cset system.site uuid e6c1838c-d5b1-4d0c-8c20-9405ca9991f6 -y` — définition de l'UUID du site
- `drush entity:delete shortcut` — suppression des raccourcis par défaut
- `drush cim -y` (exécuté **deux fois**) — import de la configuration depuis le répertoire sync
- `drush updb -y` — exécution des mises à jour de base de données
- `drush locale:check` et `drush locale:update` — mise à jour des traductions
- `drush adia` — indexation dans Elasticsearch
- `drush adum` — mise à jour des metatags
- `drush cr` — reconstruction des caches

## Tâches opérationnelles courantes

### Gestion des caches

```bash
# Nettoyage Patrimony
docker compose exec patrimony bin/console c:c

# Nettoyage Drupal
docker compose exec drupal vendor/bin/drush cr
```

### Accès shell aux conteneurs

```bash
# Shell Drupal
docker compose exec drupal bash

# Shell Patrimony
docker compose exec patrimony bash
```

### Commandes Drush personnalisées

Le Makefile fournit un passthrough pour les commandes Drush arbitraires via `make d-drush CMD="votre-commande"`.

### Makefile

Le Makefile propose un certain nombre de commandes utilitaires.

## Surveillance et dépannage

### Emplacements des logs

| Type de log          | Emplacement | Description                                                 |
|----------------------|---|-------------------------------------------------------------|
| **Import cron**      | `/var/log/audiodescription/cron-import.log` | Sortie d'exécution des imports quotidiens                   |
| **Envoi newsletter** | `/var/log/audiodescription/cron-newsletter.log` | Sortie d'exécution de l'envoi de la newsletter hebdomadaire |
| **Logs Docker**      | `docker compose logs <service>` | Stdout/stderr des conteneurs                                |
| **Logs Symfony**     | `/app/public/patrimony/var/log/` | Logs de l'application Symfony                               |
| **Logs Drupal**      | Base de données (table watchdog) | Accessible via `drush watchdog:show`                        |
