# Guide de développement

Ce guide fournit des informations pratiques pour les développeurs qui travaillent sur le code du Portail de l'audiodescription. 
Il couvre la mise en place de l'environnement local, les workflows courants, les conventions de code et les outils de qualité.

Pour les tâches spécifiques au frontend (SCSS, JavaScript, templates Twig), voir [Développement Frontend](8.1.0-Frontend-Development.md). 
Pour le développement PHP backend (services, importeurs, gestionnaires d'entités), voir [Développement Backend](8.2.0-Backend-Developement.md). 
Pour l'ajout de nouvelles intégrations partenaires, voir [Intégrer un nouveau partenaire](8.3.0-Adding-New-Partners-Integrations.md).

## Environnement de développement

### Prérequis

Le projet nécessite :

- Docker et Docker Compose pour les services conteneurisés
- Make pour l'automatisation des builds
- Git pour le contrôle de version

### Mise en place initiale

Démarrer l'environnement de développement avec Docker Compose :

```bash
make up
```

Cette commande démarre tous les services requis, notamment les bases de données PostgreSQL pour Patrimony et Drupal, Elasticsearch pour l'indexation de la recherche, et les serveurs web.

### Installation des dépendances

Installer les dépendances de Patrimony (Symfony) :

```bash
make pt-install
```

Cette commande exécute `composer install` dans le conteneur Patrimony et configure l'environnement de test.

Installer les dépendances Drupal et configurer le site :

```bash
make d-install
```

Cette séquence installe les packages Composer, initialise Drupal, définit l'UUID du site, importe la configuration, exécute les mises à jour de base de données et reconstruit les caches.

### Import automatisé (Cron)

La cible `cron-import` exécute le pipeline d'import (exécuté quotidiennement en production) :

```bash
make cron-import
```

Chaque commande d'import utilise `--create-movies=true` pour activer la création automatique des films.

Il est possible d'exécuter les tâches de la cible `cron-import` en 2 temps :
1. les tâches dédiées à Patrimony via la cible Makefile `pt-cron-import`, à exécuter quotidiennement à 01h05
2. les tâches dédiées à Drupal via la cible Makefile `d-cron-import`, à exécuter quotidiennement à 03h05

## Outils de qualité du code

### PHPStan (Analyse statique)

PHPStan analyse le code PHP à la recherche d'erreurs de typage et de bugs potentiels sans l'exécuter. Les fichiers de configuration définissent les niveaux d'analyse et les règles :

- Drupal : utilise l'extension `mglaman/phpstan-drupal` pour les patterns spécifiques à Drupal
- Patrimony : configuration personnalisée `phpstan.neon`

Lancer PHPStan :

```bash
make d-phpstan      # Drupal
make pt-phpstan   # Patrimony
```

### PHP Mess Detector (PHPMD)

PHPMD analyse le code à la recherche de complexité, de code inutilisé et de mauvaises pratiques. 
Il utilise des fichiers de règles `phpmd.xml` dans les deux applications.

Lancer PHPMD :

```bash
make d-phpmd      # Drupal
make pt-phpmd   # Patrimony
```

### PHP CodeSniffer (PHPCS/PHPCBF)

CodeSniffer applique les standards de code Drupal. 
L'outil `phpcbf` corrige automatiquement les violations.

Lancer CodeSniffer :

```bash
make d-phpcs      # Drupal
make pt-phpcs   # Patrimony
```

### Lancer toutes les vérifications qualité

Exécuter tous les outils de qualité en séquence :

```bash
make d-quality      # Drupal
make pt-quality   # Patrimony (si disponible)
```

## Workflows de développement

### Travailler avec les importeurs Patrimony

Les importeurs partenaires suivent un pattern cohérent utilisant l'interface `MovieImporterInterface` :

Chaque importeur :

1. Récupère les données depuis une API externe ou un CSV
2. Crée des entités `SourceMovie` avec des identifiants spécifiques au partenaire
3. Crée des entités `Solution` pour les fenêtres de disponibilité
4. Fait correspondre les enregistrements à des entités `Movie` canoniques via les identifiants externes

### Pattern de développement d'un importeur

Lors du développement ou de la modification d'un importeur :

**1. Définir la configuration partenaire** dans `patrimony/config/packages/patrimony.yaml`

**2. Implémenter la transformation des données** dans la classe importeur

**3. Créer/mettre à jour les entités** avec les gestionnaires de services appropriés

**4. Enregistrer dans `ImporterFactory`** avec le cas enum correspondant

### Migrations de base de données (Patrimony)

Après modification des entités Doctrine, générer et appliquer les migrations :

```bash
make pt-migrate-diff   # Générer la migration
make pt-migrate        # Appliquer la migration
```

Pour repartir de zéro en développement :

```bash
make pt-schema-reset
```

Cette commande supprime la base de données, la recrée et exécute toutes les migrations depuis le début.

### Gestion de la configuration Drupal

La configuration Drupal est stockée dans `drupal/config/sync/` et versionnée avec Git. 
Après avoir effectué des modifications de configuration dans l'interface :

1. Exporter la configuration :

```bash
make drush-cex
```

2. Committer les fichiers modifiés dans `config/sync/`

3. Sur les autres environnements, importer la configuration :

```bash
make drush-cim
```

L'UUID du site est fixé à `e6c1838c-d5b1-4d0c-8c20-9405ca9991f6` pour permettre la synchronisation de la configuration entre environnements.

## Débogage

### Accéder aux shells des conteneurs

Ouvrir des shells interactifs dans les conteneurs :

```bash
make d-sh      # Conteneur Drupal
make pt-sh   # Conteneur Patrimony
```

Depuis les conteneurs, vous pouvez exécuter directement les commandes Composer, Drush ou Symfony console.

### Consulter les logs

Docker Compose agrège les logs de tous les services :

```bash
docker compose logs -f
```

## Conventions du projet

### Injection de dépendances

Les deux applications utilisent extensivement l'injection de dépendances. Les services sont définis dans :

- Patrimony : `patrimony/config/services.yaml`
- Drupal : `drupal/web/modules/custom/audiodescription/audiodescription.services.yml`

L'injection par constructeur est privilégiée.

### Synchronisation des entités

Le champ `field_code` sert de clé de synchronisation principale entre Patrimony et Drupal. 
Cela permet aux importeurs Drupal de retrouver et mettre à jour les nœuds existants lors de la synchronisation.

### Gestion des identifiants externes

L'entité `Movie` canonique stocke plusieurs identifiants externes pour la déduplication :

- `franceTvId`
- `tf1Id`
- `arteId`
- `canalVodId`
- `imdbId`
- `plurimediaId`
- `allocineId`

Ces identifiants permettent au service `MovieFetcher` de faire correspondre les films sources aux enregistrements canoniques existants.

### Pattern de configuration API

Les identifiants et endpoints des APIs partenaires sont externalisés en variables d'environnement et référencés dans `patrimony.yaml`. 
Les importeurs les injectent via `ParameterBagInterface`.
