# Gestion des erreurs

## Objectif et périmètre

Ce document explique le système de gestion des erreurs du module Audiodescription, avec un focus particulier sur l'implémentation du code de statut **410 Gone** pour les films indisponibles. 
Le système intercepte les exceptions, convertit les erreurs d'accès refusé en réponses 410 lorsque c'est approprié, et restitue des pages d'erreur personnalisées utilisant le Système de Design de l'État français (DSFR).

Pour les informations sur la gestion des erreurs liées à la recherche, voir [Recherche & Filtrage (3.4)](https://deepwiki.com/betagouv/audiodescription/3.4-search-and-filtering). 
Pour la gestion de la configuration Drupal, voir [Gestion de la configuration (4.3)](https://deepwiki.com/betagouv/audiodescription/4.3-configuration-management).


## Vue d'ensemble

Le module Audiodescription implémente un système de gestion des erreurs personnalisé qui va au-delà des pages d'erreur standard de Drupal. 
Le différenciateur clé est le code HTTP **410 Gone**, sémantiquement correct pour les films qui existaient mais ne sont plus disponibles sur les plateformes partenaires. 
Cela améliore le SEO et l'expérience utilisateur en distinguant clairement :

- **404 Not Found** : la page n'a jamais existé ou l'URL est incorrecte
- **403 Access Denied** : l'utilisateur n'a pas les permissions
- **410 Gone** : le contenu a été explicitement supprimé ou a expiré

Le système utilise des event subscribers Symfony pour intercepter les exceptions tôt dans le cycle de vie de la requête et les transformer en réponses HTTP appropriées.

## Types d'erreurs et routes

Le module définit 3 routes d'erreur :

| Nom de route | Chemin | Statut HTTP | Méthode du contrôleur |
|---|---|---|---|
| `audiodescription.error.404` | `/error-404` | 404 | `ErrorController::error404` |
| `audiodescription.error.403` | `/error-403` | 403 | `ErrorController::error403` |
| `audiodescription.error_410` | `/system/410` | 410 | `Error410Controller::show410` |

Chaque route possède un template Twig correspondant enregistré dans le système de thème (voir [Module Audiodescription](5.1.0-Audiodescription-Module.md)).

## Les erreurs 410 Gone

### Conditions de déclenchement

L'erreur 410 est déclenchée lorsque des utilisateurs tentent d'accéder à des **films dépubliés**. Cela survient quand :

1. La disponibilité n'est plus disponible sur aucune plateforme partenaire
2. Le processus de synchronisation quotidien (voir [Importers (synchronisation avec Patrimony)](5.1.2-Importers.md)) dépublie le nœud
3. Un utilisateur visite l'URL du film ou l'a mise en favori

### Architecture des event subscribers

Le système de gestion des erreurs utilise 2 event subscribers avec des priorités différentes :

- **`MovieExceptionSubscriber`** — priorité **+200** : s'exécute en premier, intercepte les `AccessDeniedHttpException` et les convertit en réponses 410
- **`Error410ResponseSubscriber`** — priorité **-50** : s'exécute en dernier, intercepte les exceptions 410 non encore traitées

La haute priorité de `MovieExceptionSubscriber` (200) garantit qu'il s'exécute avant les gestionnaires d'accès standard de Drupal. 
La priorité basse de `Error410ResponseSubscriber` (-50) garantit qu'il s'exécute tardivement, capturant les exceptions 410 non encore gérées.

### MovieExceptionSubscriber

Ce subscriber intercepte les exceptions d'accès refusé spécifiquement pour les nœuds films dépubliés et les convertit en réponses 410.

**Vérifications effectuées** :

1. **Vérification du type d'exception** : ne traite que les `AccessDeniedHttpException`
2. **Extraction du nœud** : récupère le nœud depuis les attributs de la requête ou les paramètres de route
3. **Validation du type de contenu** : vérifie que le bundle du nœud est `movie`
4. **Statut de publication** : confirme que le nœud est dépublié (`!$node->isPublished()`)

Lorsque toutes les conditions sont remplies, il construit un render array avec le thème `error_410` et définit le statut HTTP à 410.

**Dépendances** : `@current_route_match` et `@renderer`

### Error410ResponseSubscriber

Ce subscriber gère les exceptions 410 directes qui peuvent être levées ailleurs dans le système.

**Logique d'implémentation** :

Ce subscriber adopte une approche plus simple que `MovieExceptionSubscriber`. 
Il vérifie si l'exception est une `HttpException` avec le code de statut 410, puis restitue le template `error_410` directement, sans utiliser le renderer de page complet — ce qui produit une réponse plus légère.

**Passage des données du nœud** : le subscriber cherche un attribut `error_410_node` dans la requête, qui peut être défini par le code levant l'exception 410 pour fournir du contexte.

## Error410Controller

Le contrôleur fournit une route de secours pour afficher les erreurs 410 via une navigation directe vers `/system/410`.

**Transfert de données par session** :

1. Récupère `error_410_node_data` depuis la session
2. Extrait le titre et le message si disponibles
3. Construit le render array du thème `error_410`
4. Vide les données de session pour éviter les informations périmées
5. Retourne le tableau de build (Drupal définit le statut 410 en fonction de la route)

Ce pattern permet à d'autres parties du système de rediriger vers la page 410 tout en transmettant des données contextuelles via la session.

## Visibilité des blocs sur les pages d'erreur

Le module contrôle quels blocs s'affichent sur les pages d'erreur via l'implémentation de `hook_block_view_alter()`.

### Blocs masqués

Sur les routes `audiodescription.error.403` et `audiodescription.error.404`, certains blocs sont supprimés. 
Cela crée une expérience de page d'erreur plus épurée en supprimant les éléments de navigation qui seraient confusants dans un contexte d'erreur.

### Invalidation du cache du fil d'Ariane

Le bloc fil d'Ariane a son cache désactivé (`max-age = 0`) pour garantir que les builders de fil d'Ariane dynamiques s'exécutent toujours, ce qui est nécessaire pour une navigation contextuelle précise (voir [Breadcrumb Builders (3.6.1)](https://deepwiki.com/betagouv/audiodescription/3.6.1-breadcrumb-builders)).

## Résumé du flux de gestion des erreurs

Les 3 points d'entrée convergent tous vers le thème `error_410` :

| Point d'entrée | Composant | Mécanisme |
|---|---|---|
| Accès à un nœud film dépublié | `MovieExceptionSubscriber` (priorité +200) | Intercepte `AccessDeniedHttpException`, vérifie le bundle et le statut de publication, restitue via `main_content_renderer.html` |
| Exception 410 levée directement | `Error410ResponseSubscriber` (priorité -50) | Intercepte `HttpException` avec statut 410, restitue le template directement |
| Navigation directe vers `/system/410` | `Error410Controller` | Récupère les données de session, construit le render array, vide la session |

Les 3 convergent vers le template `error-410.html.twig` du thème `ad_theme`, qui produit une réponse HTTP 410 sémantiquement correcte en utilisant les composants DSFR.

## Configuration et dépendances

### Définitions des services

Les 2 event subscribers sont enregistrés dans `audiodescription.services.yml`.

**Dépendances clés :**

- `current_route_match` : extraire le nœud de la route courante
- `renderer` : restituer les render arrays en HTML
- `main_content_renderer.html` : rendu de page complet avec mise en page (injecté dans le constructeur de `MovieExceptionSubscriber`)

### Configuration du bloc titre de page

Le bloc de titre de page est configuré pour se masquer sur certaines pages. 
Le `hook_block_view_alter()` du module fournit un contrôle dynamique supplémentaire pour les pages d'erreur.
