# Gestion de la configuration Drupal

Ce document explique comment la configuration Drupal est gérée, stockée et déployée dans le portail audiodescription. 
La gestion de la configuration couvre tous les paramètres du site, les types de contenu, les vues, les blocs et les configurations de modules via des fichiers YAML versionnés. 
Cela permet des déploiements cohérents entre environnements et un suivi traçable des changements de configuration.

Pour les informations sur le déploiement Docker, voir [Configuration Docker](7.1.0-Docker-Setup.md). 
Pour la planification des tâches automatisées, voir [Cron jobs et automatisation](7.2.0-Cron-Jobs-Automation.md).

## Vue d'ensemble du système de gestion de configuration

L'application utilise la **Configuration Management Initiative (CMI) de Drupal**, qui sépare la configuration du contenu. 

La configuration comprend :

- Les définitions de types de contenu
- Les configurations de champs
- Les définitions de vues
- Les placements de blocs
- Les paramètres de modules
- Les vocabulaires de taxonomie
- Les rôles et permissions des utilisateurs

Toute la configuration est stockée comme fichiers YAML versionnables et déployables entre environnements.

## Structure de stockage de la configuration

Les fichiers de configuration sont stockés dans `drupal/config/sync/` sous forme de YAML. 
Chaque objet de configuration a son propre fichier avec une convention de nommage incluant le type et l'ID.

## Config Split pour la gestion par environnement

L'application utilise le module **Config Split** pour gérer les configurations spécifiques aux environnements, permettant à certaines configurations de différer entre développement et production tout en maintenant une configuration de base partagée.

**Configurations typiques par environnement :**

*Développement* :
- `devel` — outils d'aide au développement
- `dblog` — journalisation en base de données pour le débogage
- Rapport d'erreurs verbeux

*Production* :
- `syslog` — intégration syslog
- Optimisations de performance
- Rapport d'erreurs restreint

## Workflow import/export de configuration

### Exporter la configuration

La configuration est exportée de la base de données active vers les fichiers YAML via Drush :

```bash
drush cex
# ou
drush config:export
```

Cette commande écrit toute la configuration active depuis la base de données vers `drupal/config/sync/`. 
Les changements peuvent ensuite être commités dans le contrôle de version.

### Importer la configuration

La configuration est importée depuis les fichiers YAML vers la base de données active, typiquement après avoir récupéré des changements depuis Git :

```bash
drush cim
# ou
drush config:import
```

### Statut de la configuration

Vérifier les différences entre la base de données et les fichiers YAML :

```bash
drush cst
# ou
drush config:status
```

Cette commande montre quelles configurations ont été modifiées en base de données mais pas encore exportées, ou vice-versa.

## Processus de déploiement

Les changements de configuration sont déployés dans le cadre du workflow standard de déploiement. La séquence typique :

```bash
# 1. Récupérer le dernier code
git pull

# 2. Importer la configuration
drush cim

# 3. Mettre à jour le schéma de base de données
drush updb

# 4. Reconstruire le cache
drush cr
```

Cette séquence est typiquement automatisée dans les scripts de déploiement ou les pipelines CI/CD.

## Gestion de la configuration dans le développement de modules

Les modules custom comme `audiodescription` définissent leur configuration par défaut dans leurs répertoires `config/install/`. 
Cette configuration est automatiquement importée lors de la première installation du module.

### Configuration du module custom

Le module `audiodescription` fournit des configurations par défaut pour :

* **Des hooks de thème (`audiodescription.module`)** définissent les templates pour `homepage`, `genres_list`, `movie_search` avec filtres, templates de pages d'erreur (403, 404, 410) et templates de newsletter.

* **Des configurations de blocs** pour définir les règles de visibilité et paramètres de placement.

### Configuration dans le code

Certaines configurations sont définies programmatiquement dans les services et doivent être mises à jour via le code plutôt que l'interface :

**Définitions de services :**
- `audiodescription.services.yml` : conteneurs d'injection de dépendances, entity managers, importeurs, builders de fil d'ariane

**Définitions de routes :**
- `audiodescription.routing.yml` : routes personnalisées comme `/homepage` et `/genres`

Ces configurations sont déployées via les mises à jour de code plutôt que par la synchronisation de config.

## Bonnes pratiques

### Toujours exporter après des changements

Après des modifications de configuration via l'interface Drupal :

```bash
drush cex && git add drupal/config/sync/ && git commit -m "chore: export configuration changes"
```

### Ne jamais éditer les fichiers YAML directement

Les fichiers YAML de configuration doivent être générés par Drupal, pas édités manuellement. 
Les éditions manuelles peuvent rompre des dépendances, créer une configuration invalide ou provoquer des erreurs d'import.

### Gérer les conflits de merge avec soin

En cas de conflits de configuration dans Git :

1. Importer la configuration en base locale : `drush cim`
2. Résoudre le conflit en acceptant une version
3. Ré-exporter pour garantir la cohérence : `drush cex`
