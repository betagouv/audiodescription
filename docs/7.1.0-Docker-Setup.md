# Configuration Docker

Ce document explique la configuration Docker Compose utilisée pour orchestrer le Portail de l'audiodescription en production. 
Il couvre les définitions de services, la mise en réseau, la gestion des volumes et la configuration du routage Traefik.

Pour les tâches d'import automatisées et leur planification, voir [Cron jobs et automatisation](7.2.0-Cron-Jobs-Automation.md). 
Pour la gestion de la configuration par environnement, voir [Gestion de la configuration Drupal](7.3.0-Drupal-Config-Management.md).

## Vue d'ensemble

L'application utilise Docker Compose pour orchestrer 4 services principaux en production :

- **patrimony** — backend Symfony avec serveur web Caddy
- **drupal** — CMS Drupal frontend avec serveur web Caddy
- **db** — serveur de base de données PostgreSQL 17
- **elasticsearch** — moteur de recherche Elasticsearch 8.14.3

Tous les services sont définis dans `compose.prod.yml` et communiquent via un réseau Docker interne. Le trafic externe est routé via le reverse proxy Traefik grâce à une configuration par labels.

## Service Patrimony

Le service `patrimony` exécute l'application Symfony backend responsable de l'import des données et de la canonicalisation. Il est construit depuis un Dockerfile personnalisé et utilise Caddy comme serveur web.

### Configuration

| Propriété | Valeur | Description |
|---|---|---|
| Contexte de build | `.` | Répertoire racine |
| Dockerfile | `./docker/patrimony/prod/php/Dockerfile` | Image PHP+Caddy personnalisée |
| Répertoire de travail | `/app/public/patrimony` | Racine de l'application Symfony |
| Mapping de port | `8081:80` | Port interne 80 exposé sur le port hôte 8081 |
| Politique de redémarrage | `always` | Redémarrage en cas d'échec |
| Réseaux | `ad-network`, `web` | Réseau interne + réseau Traefik |

### Volumes montés

Le service monte 3 volumes critiques :

1. **Code applicatif** : `.:/app/public` — tout le dépôt monté dans `/app/public`
2. **Configuration Caddy** : `./docker/patrimony/prod/caddy/Caddyfile:/etc/caddy/Caddyfile`
3. **Configuration PHP** : `./docker/patrimony/prod/php/php.ini:/usr/local/etc/php/php.ini`

### Labels Traefik

Les labels Traefik configurent le routage automatique. 
Le trafic de `patrimony.audiodescription.beta.gouv.fr` est routé vers ce service via HTTPS.

## Service Drupal

Le service `drupal` exécute le CMS Drupal qui fournit le site web public. Comme Patrimony, il utilise PHP avec Caddy.

### Configuration

| Propriété | Valeur | Description |
|---|---|---|
| Contexte de build | `.` | Répertoire racine |
| Dockerfile | `./docker/drupal/prod/php/Dockerfile` | Image PHP+Caddy personnalisée |
| Répertoire de travail | `/app/public/drupal` | Racine de l'installation Drupal |
| Mapping de port | `8080:80` | Port interne 80 exposé sur le port hôte 8080 |
| Politique de redémarrage | `always` | Redémarrage en cas d'échec |
| Réseaux | `ad-network`, `web` | Réseau interne + réseau Traefik |
| Dépendances | `db` | Attend le démarrage du service base de données |

### Volumes montés

Identiques à Patrimony :

1. **Code applicatif** : `.:/app/public`
2. **Configuration Caddy** : `./docker/drupal/prod/caddy/Caddyfile:/etc/caddy/Caddyfile`
3. **Configuration PHP** : `./docker/drupal/prod/php/php.ini:/usr/local/etc/php/php.ini`

### Labels Traefik

Le service Drupal gère à la fois le domaine principal et le sous-domaine www. 
Le trafic de `audiodescription.beta.gouv.fr` ET `www.audiodescription.beta.gouv.fr` est routé vers ce service, avec le middleware `strip-www` qui supprime le préfixe www.

## Service Base de données

Le service `db` exécute PostgreSQL 17 et héberge les bases de Patrimony et Drupal.

### Configuration

| Propriété | Valeur | Description |
|---|---|---|
| Image | `postgres:17-bookworm` | PostgreSQL 17 officiel sur Debian Bookworm |
| Politique de redémarrage | `always` | Redémarrage en cas d'échec |
| Réseau | `ad-network` | Réseau interne uniquement |
| Volume | `db-data:/var/lib/postgresql/data` | Stockage persistant des fichiers de base de données |

### Variables d'environnement requises

- `POSTGRES_DB` — nom de la base de données initiale
- `POSTGRES_USER` — nom du superutilisateur PostgreSQL
- `POSTGRES_PASSWORD` — mot de passe du superutilisateur

Ces variables sont lues depuis l'environnement hôte.

### Script d'initialisation

Le service monte un script d'initialisation exécuté au premier démarrage qui crée la base de données Patrimony séparée lors de l'initialisation du conteneur.

## Service Elasticsearch

Le service `elasticsearch` fournit les capacités de recherche plein texte pour le frontend Drupal.

### Configuration

| Propriété | Valeur | Description |
|---|---|---|
| Image | `docker.elastic.co/elasticsearch/elasticsearch:8.14.3` | Elasticsearch 8.14.3 officiel |
| Politique de redémarrage | `always` | Redémarrage en cas d'échec |
| Réseau | `ad-network` | Réseau interne uniquement |
| Volume | `elastic-data:/usr/share/elasticsearch/data` | Stockage persistant des indices |

### Variables d'environnement

- `discovery.type=single-node` — mode cluster mono-nœud
- `xpack.security.enabled=false` — désactivation de la sécurité X-Pack (réseau interne uniquement)
- `ES_JAVA_OPTS=-Xms2048m -Xmx2048m` — tas JVM fixé à 2 Go

Les 2 Go de tas garantissent une mémoire suffisante pour l'indexation et la recherche des films.

## Configuration réseau

La configuration Docker utilise 2 réseaux.

### ad-network (réseau interne)

Créé par Docker Compose avec le driver par défaut. 

Les 4 services se connectent à ce réseau pour la communication inter-services :
- `patrimony` / `db` : lecture/écriture de la base Patrimony
- `drupal` / `db` : lecture/écriture de la base Drupal
- `drupal` / `elasticsearch` : requêtes de recherche

Ce réseau est isolé de l'accès externe.

### web (réseau externe)

Réseau externe géré par Traefik. 
Seuls `patrimony` et `drupal` se connectent à ce réseau pour recevoir le trafic routé par Traefik. 

**Ce réseau doit exister avant de démarrer les services**.

## Gestion des volumes

### Volumes nommés

| Nom du volume | Service | Point de montage | Rôle |
|---|---|---|---|
| `db-data` | db | `/var/lib/postgresql/data` | Fichiers de données PostgreSQL |
| `elastic-data` | elasticsearch | `/usr/share/elasticsearch/data` | Indices Elasticsearch |

Les volumes nommés sont gérés par Docker et persistent au-delà du cycle de vie des conteneurs.

### Bind Mounts

Les services `patrimony` et `drupal` montent l'ensemble du dépôt dans `/app/public`, leur permettant d'accéder à leurs répertoires applicatifs respectifs. 
Les fichiers de configuration sont montés pour surcharger les paramètres par défaut dans les conteneurs.

## Intégration Traefik

L'application s'appuie sur le reverse proxy Traefik pour la terminaison HTTPS et le routage. 
Traefik doit être déployé dans une stack Docker séparée et connecté au réseau `web`.

### Routage par labels

**Service Patrimony :**
- Nom du router : `audiodescription_patrimony`
- Règle d'hôte : `patrimony.audiodescription.beta.gouv.fr`
- Point d'entrée : `websecure` (HTTPS)
- Middleware : `strip-www`

**Service Drupal :**
- Nom du router : `audiodescription`
- Règle d'hôte : `audiodescription.beta.gouv.fr` OU `www.audiodescription.beta.gouv.fr`
- Point d'entrée : `websecure` (HTTPS)
- Middleware : `strip-www`

### Middleware `strip-www`

Le middleware `strip-www` référencé dans les deux services doit être défini dans la configuration Traefik. 
Il redirige le trafic du sous-domaine www vers le domaine apex.

### Mapping des ports

Bien que les labels Traefik gèrent le routage en production, les deux services exposent aussi des ports directement pour le débogage : Patrimony sur `8081`, Drupal sur `8080`. 
En production, tout le trafic doit transiter par Traefik plutôt que par ces ports exposés.

## Configuration d'environnement

### Variables d'environnement Docker Compose

Ces variables doivent être définies dans un fichier `.env` dans le même répertoire que `compose.prod.yml` ou exportées dans le shell avant d'exécuter `docker compose up` :

| Variable | Service | Rôle |
|---|---|---|
| `POSTGRES_DB` | db | Nom de la base de données initiale (typiquement `audiodescription`) |
| `POSTGRES_USER` | db | Superutilisateur PostgreSQL |
| `POSTGRES_PASSWORD` | db | Mot de passe du superutilisateur PostgreSQL |

### Variables d'environnement des applications

Des variables supplémentaires sont requises par les applications Patrimony et Drupal elles-mêmes, configurées dans leurs fichiers respectifs :

- **Patrimony** : `.env` dans le répertoire patrimony
- **Drupal** : `settings.php` ou `settings.local.php`

Ces fichiers configurent les connexions à la base de données, les clés d'API et autres paramètres runtime.

## Démarrage des services

```bash
docker compose -f compose.prod.yml up -d --build --remove-orphans -t 0
```

Cette commande :

1. Construit les images si les Dockerfiles ont changé
2. Crée les réseaux s'ils n'existent pas (sauf `web` qui doit préexister)
3. Crée les volumes s'ils n'existent pas
4. Démarre les conteneurs dans l'ordre de dépendance (`db` → `drupal`)
5. S'exécute en mode détaché (`-d`)

### Dépendances de démarrage

La directive `depends_on` garantit l'ordre de démarrage correct : `drupal` attend que `db` soit créé.

> **Note :** `depends_on` attend uniquement la création du conteneur, pas la disponibilité complète du service. 
> L'application Drupal inclut une logique de reconnexion pour gérer le délai d'initialisation de PostgreSQL.

### Comportement de redémarrage

Tous les services utilisent `restart: always`, garantissant leur redémarrage automatique en cas de défaillance du conteneur, après un redémarrage du serveur, ou après un redémarrage du daemon Docker.
