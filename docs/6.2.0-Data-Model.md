# Modèle de données

Ce document décrit le modèle de données du backend Patrimony, implémenté avec Symfony et Doctrine ORM.
La base de données Patrimony constitue la **source de vérité** pour les données films agrégées depuis plusieurs partenaires de contenu.

Pour les informations sur la façon dont les données sont importées dans ces entités, voir [Transformation et correspondance des données](6.1.2-Data-Transformation-Matching.md). 

Pour les détails sur la synchronisation de ce modèle vers Drupal, voir [Importers de synchronisation Patrimony (3.1.2)]().

Le modèle de données est organisé en trois groupes logiques : les **entités canoniques** (films dédupliqués), les **entités sources** (données brutes partenaires), et les **entités de configuration** (partenaires et types d'offres).


## Entités canoniques

Les entités canoniques représentent les enregistrements films faisant autorité, dédupliqués, qui agrègent les données de plusieurs sources.

### Entité Movie

L'entité `Movie` est l'enregistrement canonique central qui représente un film unique. Elle stocke les métadonnées consolidées et maintient les identifiants externes de tous les partenaires de contenu.

**Propriétés principales :**

| Propriété | Type | Description |
|---|---|---|
| `id` | int | Clé primaire |
| `code` | string | Code CNC unique (clé de synchronisation) |
| `title` | string | Titre du film |
| `hasAd` | boolean | Présence d'une audiodescription |
| `productionYear` | int | Année de production |
| `synopsis` | text | Description du film |
| `poster` | text | URL de l'affiche |
| `duration` | int | Durée en minutes |

**Identifiants externes :** l'entité `Movie` maintient un champ nullable distinct pour l'ID interne de chaque partenaire, permettant la correspondance inter-partenaires :

| Champ | Partenaire |
|---|---|
| `cncId` | Registre CNC |
| `arteId` | Arte TV |
| `canalVodId` | Canal+ VOD |
| `orangeVodId` | Orange VOD |
| `allocineId` | AlloCiné |
| `laCinetekId` | LaCinetek |
| `franceTvId` | France TV |
| `tf1Id` | TF1 |
| `imdbId` | IMDB |
| `plurimediaId` | Plurimedia |
| `isanId` | Registre ISAN |

**Relations :**

- **ManyToMany avec Director** 
- **ManyToMany avec Genre**
- **ManyToMany avec Nationality**
- **ManyToOne avec PublicRestriction**
- **OneToMany avec ActorMovie**
- **OneToMany avec Solution**
- **OneToMany avec SourceMovie**

**Trait Timestampable :** l'entité utilise le trait `Timestampable` pour gérer automatiquement les champs `createdAt` et `updatedAt` via les lifecycle callbacks Doctrine.

### Entité Director

Représente les réalisateurs de films avec un stockage normalisé des noms.

**Propriétés :**

- `id` (int, PK)
- `fullname` (string, unique) — nom complet tel qu'affiché
- `firstname` (string, nullable) — composante prénom
- `lastname` (string, nullable) — composante nom de famille

Le champ `fullname` sert d'identifiant unique et est utilisé dans les requêtes films

### Entité Genre

Représente les genres cinématographiques avec support hiérarchique via une relation `mainGenre` optionnelle.

**Propriétés :**

- `id` (int, PK)
- `name` (string, unique)
- `mainGenre` (Genre, nullable, auto-référentiel) — genre parent pour la catégorisation

Les genres peuvent avoir des sous-genres. La newsletter utilise cette hiérarchie pour afficher les noms de genres consolidés.

### Entité Nationality

Représente les pays de production des films.

**Propriétés :**

- `id` (int, PK)
- `name` (string, unique)

### Entité PublicRestriction

Représente les classifications par tranche d'âge.

**Propriétés :**

- `id` (int, PK)
- `code` (string, unique) — code de classification d'âge (ex. "10", "12", "16", "18")
- `name` (string) — nom d'affichage

### Entités Actor et ActorMovie

Le système de casting utilise un pattern d'entité de jointure pour supporter les informations de rôle.

**Actor :**

- `id` (int, PK)
- `fullname` (string, unique)
- `firstname` (string, nullable)
- `lastname` (string, nullable)

**ActorMovie (entité de jointure) :**

- `id` (int, PK)
- `movie` (Movie, ManyToOne)
- `actor` (Actor, ManyToOne)
- `role` (string, nullable) — nom du personnage

Cela permet de stocker le rôle spécifique joué par un acteur dans chaque film.

## Entités sources

Les entités sources stockent les données brutes telles que reçues des partenaires.

### Entité SourceMovie

`SourceMovie` représente un enregistrement film tel que fourni par un partenaire spécifique. 
Plusieurs enregistrements `SourceMovie` peuvent finalement pointer vers le même `Movie`.

**Propriétés principales :**

| Propriété | Type | Description |
|---|---|---|
| `id` | uuid | Clé primaire (UUID) |
| `internalPartnerId` | string | ID interne du partenaire pour ce film |
| `title` | string | Titre tel que fourni par le partenaire |
| `productionYear` | int | Année depuis les données partenaires |
| `directors` | json | Tableau d'objets réalisateurs |
| `casting` | json | Tableau d'objets acteurs avec rôles |
| `genres` | json | Tableau de noms de genres |
| `synopsis` | text | Description depuis le partenaire |
| `poster` | text | URL de l'affiche depuis le partenaire |
| `duration` | int | Durée en minutes |
| `public` | string | Code de classification d'âge |

**Structures des champs JSON :**

Le champ `directors` stocke un tableau d'objets avec la clé `fullname`. Le champ `casting` stocke les acteurs avec les informations de rôle optionnelles.

**Relations :**

- **ManyToOne avec Partner** : obligatoire — identifie la source des données
- **ManyToOne avec Movie** : nullable — défini après correspondance
- **OneToMany avec Solution** : un source movie peut avoir plusieurs fenêtres de disponibilité

**Cycle de vie :** les enregistrements `SourceMovie` sont supprimés lorsqu'ils expirent (le partenaire ne les fournit plus).

### Entité Solution

`Solution` représente la disponibilité temporalisée d'un film sur une plateforme partenaire spécifique avec un type d'offre spécifique.

**Propriétés principales :**

| Propriété | Type | Description |
|---|---|---|
| `id` | uuid | Clé primaire (UUID) |
| `internalPartnerId` | string | ID contenu du partenaire |
| `link` | text | URL pour regarder le film |
| `condition` | text | Conditions d'accès supplémentaires |
| `startRights` | datetime | Début de la disponibilité |
| `endRights` | datetime | Expiration de la disponibilité |

**Relations :**

- **ManyToOne avec SourceMovie**
- **ManyToOne avec Partner** : obligatoire
- **ManyToOne avec Offer** : obligatoire
- **ManyToOne avec Movie** : nullable, défini après correspondance

**Logique de fenêtre de droits :** les solutions avec `startRights` et `endRights` définissent une fenêtre de disponibilité précise. Les requêtes newsletter utilisent ces dates pour trouver les films nouvellement disponibles ou en cours d'expiration.

**Cycle de vie :** les solutions sont supprimées à leur expiration ou lors de l'exécution d'un import partenaire.

## Entités de configuration partenaires

### Entité Partner

Définit les plateformes fournisseurs de contenu.

**Propriétés :**

- `id` (int, PK)
- `code` (string, unique) — code partenaire depuis l'enum `PartnerCode`
- `name` (string) — nom d'affichage

**Codes partenaires :**

- `FRANCE_TV` — France Télévisions
- `TF1` — TF1+
- `ARTE` — Arte TV
- `CANAL_VOD` — Canal+ VOD
- `CANAL_REPLAY` — Canal+ Replay
- `ORANGE_VOD` — Orange VOD
- `LACINETEK` — LaCinetek

Les partenaires sont créés automatiquement par les importers s'ils n'existent pas.

### Entité Offer

Définit les types d'accès aux contenus.

**Propriétés :**

- `id` (int, PK)
- `code` (string, unique) — code offre depuis l'enum `OfferCode`
- `name` (string) — nom d'affichage

Le type d'offre principal est `FREE_ACCESS`, utilisé pour les contenus librement accessibles.

## Correspondance et déduplication des entités

Le système utilise un processus en plusieurs étapes pour faire correspondre les enregistrements `SourceMovie` entrants aux entités `Movie`.

### Logique de la requête de correspondance

La méthode `MovieRepository::findByIds()` construit une requête dynamique qui recherche les films correspondant à n'importe quel ID externe fourni :

1. Accepte un tableau d'IDs externes et un code CNC
2. Construit des conditions OR pour chaque ID non vide
3. Retourne tous les films correspondant à au moins un identifiant

### Résolution de la correspondance

Le service `MovieFetcher` gère la résolution de la correspondance selon trois cas :

1. **Aucune correspondance** : créer un nouveau `Movie` depuis les données du `SourceMovie`
2. **Correspondance unique** : utiliser le `Movie` existant et ajouter les nouveaux IDs externes
3. **Correspondances multiples** : utiliser le premier résultat (détection de doublons potentiels nécessaire)

Le `Movie` trouvé ou nouvellement créé est ensuite lié au `SourceMovie` et à la `Solution`.

## Méthodes de requête du repository

Le `MovieRepository` fournit des requêtes spécialisées pour la génération de newsletter et le suivi de disponibilité.

### Requêtes films gratuits

**`findNewFreeMovies()`** retourne les films devenus gratuitement disponibles dans les 7 derniers jours.

Critères :
- `offer.code = 'FREE_ACCESS'`
- `solution.startRights >= (maintenant - 7 jours)`
- `solution.startRights <= maintenant`
- `solution.endRights >= (maintenant + 7 jours)`
- `solution.link IS NOT NULL`
- `movie.hasAd = true`
- Optionnel : filtrage par code partenaire

**`findNearEndFreeMovies()`** retourne les films gratuits expirant dans 3 à 15 jours.

Critères :
- `offer.code = 'FREE_ACCESS'`
- `solution.endRights BETWEEN (maintenant + 3 jours) AND (maintenant + 15 jours)`
- `solution.link IS NOT NULL`
- `movie.hasAd = true`
- Exclusion des films déjà sélectionnés
- Tri par `endRights ASC`
- Limité à `$maxResult`

**`findNotSelectedFreeMovies()`** retourne les films gratuits actuellement disponibles n'étant pas dans la liste d'exclusion.

**`countFreeMovies()`** : nombre total de films gratuits actuellement disponibles.

Ces requêtes alimentent le système de newsletter qui envoie des mises à jour hebdomadaires sur les films audiodécrits gratuits.


## Flux de données dans le modèle d'entités

Voici comment les données transitent depuis les APIs partenaires à travers le modèle d'entités :

1. **Phase d'import** : l'importer récupère les données depuis l'API ou le fichier fournit par le partenaire, parse la réponse, crée un `SourceMovie` avec les champs JSON bruts
2. **Couche source** : le `SourceMovie` est lié au `Partner` ; une `Solution` est créée avec la fenêtre de droits (facultative) et le lien
3. **Phase de correspondance** : extraction des IDs externes, requête pour trouver un `Movie` existant, résolution de la correspondance
4. **Couche canonique** : le `Movie` est alimenté avec les relations normalisées (Director, Genre, etc.)
5. **Liaison** : le `SourceMovie` et la `Solution` sont liés au `Movie` pour les requêtes

**Suivi des mises à jour :** lors de l'exécution des imports partenaires, le timestamp `updatedAt` de tous les `Movie` ayant des `Solution` de ce partenaire est rafraîchi.

## Exposition via API Platform

Les entités `Movie` et `Solution` sont exposées via API Platform pour être consommées par Drupal.

**Configuration API Movie :**

- Route : `/api/v1/movies`
- Opération : `GetCollection`
- Filtrable par `updatedAt` via `DateFilter`
- Des groupes de sérialisation contrôlent quelles relations sont incluses

**Configuration API Solution :**

- Route : `/api/v1/solutions`
- Opération : `GetCollection`

L'importer Drupal interroge ces endpoints filtrés par `updatedAt` pour ne synchroniser que les enregistrements modifiés, comme documenté dans [Importers de synchronisation Patrimony (3.1.2)]().
