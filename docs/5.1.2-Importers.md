# Importers (synchronisation avec Patrimony)

## Objectif et périmètre

Les importers permettent de synchroniser l'application Drupal avec Patrimony. 
Ce sont un ensemble de classes de services qui synchronisent les données du backend Patrimony (application Symfony) dans Drupal sous forme de nœuds et de termes de taxonomie. 
Ces importers lisent depuis l'API REST de Patrimony et créent ou mettent à jour les entités Drupal correspondantes, en maintenant le champ `field_code` comme clé de synchronisation entre les systèmes.

Cette page documente l'architecture des importers, la logique de transformation des données et le processus d'exécution. 

Pour les informations sur le backend Patrimony qui génère les données sources, voir [Patrimony](6.0-Patrimony.md).
Pour les détails sur les intégrations APIs partenaires qui alimentent Patrimony, voir [Intégrations des APIs partenaires](6.1.1-Integration-API-partners.md). 
Pour les définitions de services et l'injection de dépendances, voir [Services](5.1.1-Services.md).

## Architecture des importers

Le système fournit 7 importers spécialisés, chacun responsable d'un type d'entité différent. 
Tous les importers suivent un pattern commun : récupérer les données depuis l'API Patrimony, les transformer, et déléguer aux Entity Managers pour la persistance.

## Enregistrement des services

Tous les importers Patrimony sont enregistrés comme services Drupal avec leurs dépendances injectées.

| ID de service | Classe | Dépendances Entity Manager |
|---|---|---|
| `audiodescription.genre_patrimony_importer` | `GenrePatrimonyImporter` | `genre` |
| `audiodescription.public_patrimony_importer` | `PublicPatrimonyImporter` | `public` |
| `audiodescription.nationality_patrimony_importer` | `NationalityPatrimonyImporter` | `nationality` |
| `audiodescription.director_patrimony_importer` | `DirectorPatrimonyImporter` | `director` |
| `audiodescription.partner_patrimony_importer` | `PartnerPatrimonyImporter` | `partner` |
| `audiodescription.offer_patrimony_importer` | `OfferPatrimonyImporter` | `offer` |
| `audiodescription.movie_patrimony_importer` | `MoviePatrimonyImporter` | `movie`, `nationality`, `genre`, `director`, `public`, `partner`, `offer` |

Tous les importers reçoivent :

- `EntityTypeManager` pour accéder au stockage des entités Drupal.
- `ConfigPagesLoaderServiceInterface` pour accéder aux credentials API.
- Le ou les Entity Manager(s) spécifique(s) pour créer/mettre à jour les entités.

## Séquence d'exécution des imports

Les importers doivent s'exécuter dans un ordre précis car les films dépendent de termes de taxonomie qui doivent exister au préalable. La séquence garantit l'intégrité référentielle :

1. Genres
2. Publics
3. Nationalités
4. Réalisateurs
5. Partenaires
6. Offres
7. Films (dépend de toutes les entités précédentes)

## Implémentation du MoviePatrimonyImporter

Le `MoviePatrimonyImporter` est l'importer le plus complexe, gérant les films avec des relations imbriquées vers plusieurs types d'entités.

### Dépendances du constructeur

L'importer nécessite huit entity managers car il doit créer ou référencer des entités de plusieurs types lors de l'import des films.

### Communication avec l'API

L'importer récupère les données depuis l'API REST Patrimony via de la pagination :

| Configuration | Source | Rôle |
|---|---|---|
| `field_patrimony_url` | Page de config `patrimony` | URL de base de l'API |
| `field_patrimony_token` | Page de config `patrimony` | Bearer token d'authentification |
| `ITEMS_PER_PAGE` | Constante de classe | 200 films par requête |

Format de l'endpoint API : `/api/v1/movies?page=N&itemsPerPage=200`

La réponse suit la spécification Hydra JSON-LD avec `hydra:member` contenant les films et `hydra:view['hydra:next']` indiquant la présence de pages supplémentaires.

### Extraction et transformation des données

Pour chaque film dans la réponse API, l'importer extrait les métadonnées (titre, synopsis, année de production, affiche, IDs externes) et les solutions de disponibilité (partenaire, offre, lien, dates de droits).

### Mapping code partenaire / type d'offre

L'importer mappe les codes partenaires Patrimony vers les types d'offres via un switch :

| Code partenaire | Type d'offre | Description |
|---|---|---|
| `ARTE` | `FREE_ACCESS` | Streaming gratuit |
| `FRANCE_TV` | `FREE_ACCESS` | Streaming gratuit |
| `TF1` | `FREE_ACCESS` | Streaming gratuit |
| `ORANGE_VOD` | `TVOD` | Vidéo à la demande transactionnelle |
| `LACINETEK_TVOD` | `TVOD` | Vidéo à la demande transactionnelle |
| `CANAL_VOD` | `TVOD` | Vidéo à la demande transactionnelle |
| `LACINETEK_SVOD` | `SVOD` | Vidéo à la demande par abonnement |
| `CANAL_REPLAY` | `SVOD` | Vidéo à la demande par abonnement |

Ce mapping regroupe les solutions par type d'offre avant de créer la structure de paragraphes imbriqués.

### Traitement du synopsis

Le champ synopsis subit un décodage des entités HTML et une suppression des balises pour garantir un texte brut propre via `html_entity_decode(strip_tags())`. 
Cela empêche les balises HTML d'apparaître dans le synopsis affiché.

## Création d'entités par le MovieManager

Le `MovieManager` gère la logique complexe de création ou mise à jour des nœuds films avec leurs structures de paragraphes imbriqués.

### Mapping des champs du nœud

| Champ Patrimony | Champ Drupal | Type |
|---|---|---|
| `code` | `field_code` | Chaîne (clé de sync) |
| `title` | `title` | Chaîne |
| `hasAd` | `field_has_ad` | Booléen |
| `productionYear` | `field_production_year` | Chaîne |
| `synopsis` | `field_synopsis` | Texte avec format |
| `allocineId` | `field_allocine_id` | Chaîne |
| `arteId` | `field_arte_id` | Chaîne |
| `canalVodId` | `field_canal_vod_id` | Chaîne |
| `franceTvId` | `field_france_tv_id` | Chaîne |
| `laCinetekId` | `field_lacinetek_id` | Chaîne |
| `orangeVodId` | `field_orange_vod_id` | Chaîne |
| `poster` | `field_poster_external` | Chaîne (URL) |

Le champ `field_code` sert d'identifiant unique pour retrouver les films existants.

### Recherche des films existants

Le manager interroge Drupal via `field_code` pour déterminer si un film existe déjà :

- Si aucun film trouvé, alors création d'un nouveau nœud
- Si un film trouvé, alors on le met à jour
- Si plusieurs films sont trouvés, alors on loggue un bug (ça ne devrait jamais arriver)

### Nettoyage et recréation des paragraphes

Avant la mise à jour d'un film, le manager supprime tous les paragraphes offre et partenaire existants pour éviter les orphelins. 
Cela garantit une ardoise vierge avant de recréer la structure de paragraphes depuis les dernières données Patrimony.

## Structure de paragraphes imbriqués

La relation film ↔ disponibilité partenaire utilise une structure à trois niveaux :

```
Movie (nœud)
└── pg_offer (paragraphe offre — ex. "Gratuit", "SVOD")
    └── pg_partner (paragraphe partenaire — ex. "France TV", lien, dates de droits)
```

### Gestion des dates

Les dates de droits provenant de Patrimony sont converties du format ISO 8601 vers le format de date Drupal. Cela permet aux templates Drupal de déterminer si un film est actuellement disponible sur une plateforme partenaire.

## Implémentation du PartnerManager

Le `PartnerManager` crée ou met à jour les termes de taxonomie partenaire via une recherche par code.

### Mapping des champs

| Clé d'entrée | Champ taxonomie | Rôle |
|---|---|---|
| `code` | `field_taxo_code` | Identifiant unique (ex. "FRANCE_TV") |
| `name` | `name` | Nom d'affichage (ex. "France Télévisions") |
| `pronunciation` | `field_taxo_pronunciation` | Guide de prononciation pour l'accessibilité |
| `displayOrder` | `field_taxo_order` | Ordre de tri pour l'affichage |

Si aucun nom n'est fourni, le code est utilisé comme nom du terme.

## Exécution via commande Drush

L'`ImportAllPatrimonyCommand` fournit l'interface Drush pour exécuter les imports.

### Définition de la commande

La commande est accessible via deux alias :

- `drush ad:import:all`
- `drush adia`

### Flux d'exécution de l'import

⚠️ La plupart des importers de taxonomie sont commentés dans l'implémentation actuelle, seul l'import des films étant actif.

### Indexation post-import

Après l'import des films, la commande déclenche automatiquement l'indexation Elasticsearch. 
Cela garantit que l'index de recherche est immédiatement mis à jour avec les films nouveaux ou modifiés.


## Gestion du cache

Le `MoviePatrimonyImporter` vide les définitions mises en cache de l'entity type manager après chaque page et à la fin de l'import. 
Cela empêche les métadonnées d'entités obsolètes de causer des problèmes lors d'imports de longue durée portant sur des centaines d'entités.

## Gestion des erreurs et logging

L'importer utilise le `LoggerAwareTrait` pour la journalisation et intercepte les exceptions HTTP. 
Les erreurs lors de la communication API sont journalisées mais n'arrêtent pas l'ensemble du processus d'import.

## Résumé

Les importers de synchronisation Patrimony fournissent la couche de synchronisation critique entre le backend Patrimony et le frontend Drupal. 
Le `MoviePatrimonyImporter` orchestre la création de structures imbriquées complexes en :

1. Récupérant les données paginées depuis l'API REST Patrimony
2. Créant/mettant à jour les termes de taxonomie pour les genres, réalisateurs, partenaires et offres
3. Mappant les codes partenaires vers les types d'offres
4. Construisant des structures de paragraphes à trois niveaux (Movie → `pg_offer` → `pg_partner`)
5. Préservant les IDs externes pour les références inter-systèmes
6. Déclenchant la réindexation de la recherche

Tous les importers utilisent des Entity Managers pour abstraire les opérations CRUD et garantir une gestion cohérente des données entre les types d'entités.
