# Configuration Search API

## Objectif et périmètre

Ce document décrit la configuration du module Search API et son intégration avec Elasticsearch pour la fonctionnalité de recherche de films. 
Il couvre les dépendances des modules, la configuration de l'index `movies`, le mapping des champs et la configuration du connecteur Elasticsearch.

Pour les informations sur la construction des requêtes et l'utilisation du service search manager, voir [Moteur de recherche et système de requêtage](5.4.2-Search-Manager-Query-Building.md). 
Pour les détails sur l'implémentation du formulaire de recherche, voir [Formulaire de recherche et filtres)](5.4.1-Search-Fom-Filters.md).

## Architecture Search API

L'infrastructure de recherche utilise le module Search API de Drupal comme couche d'abstraction entre l'application et Elasticsearch. 
Cette architecture fournit une interface standardisée pour l'indexation et les requêtes, tout en permettant de changer le backend de recherche sous-jacent si nécessaire.

## Dépendances des modules

La fonctionnalité de recherche nécessite 3 packages Composer principaux installés dans l'application Drupal :

| Package | Version | Rôle |
|---|---|---|
| `drupal/search_api` | ^1.35 | Framework Search API de base |
| `drupal/search_api_elasticsearch` | dev-7.x-1.x | Plugin backend Elasticsearch |
| `drupal/elasticsearch_connector` | ^8.0@alpha | Wrapper client Elasticsearch |

Ces modules sont activés dans l'installation Drupal.

## L'index Movies

Le système définit un unique index Search API nommé `movies` qui indexe tous les nœuds films. 
L'index est chargé et utilisé dans l'ensemble du système de recherche.

### Chargement et initialisation de l'index

Le service `MovieSearchManager` charge l'index lors de son initialisation.

### Champs indexés

L'index `movies` suit les champs des nœuds films et de leurs entités liées. Les opérations de requête supportées sont :

- **Recherche plein texte** sur `field_title` et `field_synopsis` pour les requêtes par mots-clés
- **Correspondance exacte** sur `field_code` pour la synchronisation Patrimony
- **Filtrage booléen** sur `field_has_ad` pour la disponibilité de l'audiodescription
- **Requêtes par plage** sur `field_production_year` pour le filtrage temporel
- **Filtrage multi-valeurs** sur `field_genres` et `field_pg_partner` pour la recherche à facettes

## Construction des requêtes avec l'index

Le `MovieSearchManager` utilise l'index chargé pour construire les requêtes de recherche. 
Chaque opération de requête s'appuie sur le query builder de l'index.

## Configuration du connecteur Elasticsearch

Le connecteur Elasticsearch doit être configuré pour pointer vers le service Elasticsearch. 
La configuration est typiquement stockée dans les fichiers `elasticsearch_connector.cluster.*.yml`.

### Paramètres de connexion

La configuration du connecteur inclut :

- **Hôte** : `elasticsearch` (nom du service Docker) ou `localhost` pour le développement local
- **Port** : `9200` (port Elasticsearch par défaut)
- **Protocole** : HTTP ou HTTPS
- **Authentification** : identifiant/mot de passe optionnel ou clé API
- **Timeout** : timeouts de connexion et de requête

## Opérations sur l'index

### Indexation du contenu

Le contenu est indexé dans Elasticsearch via plusieurs mécanismes : automatiquement lors de la création/mise à jour des nœuds, et manuellement via la commande Drush de réindexation.

### Surveillance de l'état de l'index

Les administrateurs peuvent surveiller l'état de l'index via l'interface d'administration Drupal à `/admin/config/search/search-api/index/movies`. 

Cette interface affiche :

- **Éléments indexés** : nombre d'éléments indexés avec succès
- **Éléments à indexer** : nombre d'éléments en attente dans la file
- **Statut** : santé globale de l'index
- **Dernière indexation** : horodatage de la dernière opération

## Intégration avec MovieSearchManager

Le service `MovieSearchManager` encapsule toutes les interactions avec l'index Search API pour la fonctionnalité de recherche de films.

Ce pattern de service centralisé garantit :

- Un point d'accès unique à l'index
- Une construction cohérente des requêtes
- Une logique de recherche réutilisable entre contrôleurs et formulaires
- Une maintenance simplifiés

## Détails du mapping des champs

L'index Search API mappe les données des champs Drupal vers des types compatibles Elasticsearch :

| Champ Drupal | Type | Indexé | Stocké | Rôle |
|---|---|---|---|---|
| `title` | text | Oui | Oui | Recherche plein texte, affichage |
| `field_code` | string | Oui | Oui | Correspondance exacte, clé de sync |
| `field_synopsis` | text | Oui | Non | Recherche plein texte |
| `field_production_year` | integer | Oui | Oui | Filtrage par plage, affichage |
| `field_has_ad` | boolean | Oui | Oui | Filtrage booléen |
| `field_genres` | integer | Oui | Oui | Filtrage multi-valeurs |
| `field_pg_partner` | integer | Oui | Oui | Filtrage multi-valeurs |
| `nid` | integer | Oui | Oui | Chargement des entités |
| `status` | boolean | Oui | Non | Filtre de publication |

La configuration de l'index peut spécifier :

- **Valeurs de boost** pour le réglage de la pertinence
- **Processeurs** pour la transformation des données (tokenisation, stemming, etc.)
- **Sources de données** (typiquement `entity:node`)
- **Paramètres spécifiques au backend** (paramètres d'index Elasticsearch, mappings)

## Considérations de performance

### Optimisation de l'index

L'index movies bénéficie de plusieurs stratégies d'optimisation :

1. **Indexation sélective des champs** : seuls les champs utilisés en recherche ou filtrage sont indexés
2. **Limitation de la taille des résultats** : les requêtes utilisent `->range()` pour limiter l'ensemble à 20 éléments
3. **Groupement des conditions** : les conditions multiples utilisent des groupes OR pour optimiser la structure des requêtes
4. **Chargement des entités** : les résultats sont chargés comme entités complètes uniquement après filtrage

### Stratégie de cache

Les résultats de recherche incluent des métadonnées de cache pour contrôler le comportement de mise en cache. 
Cela garantit que les utilisateurs voient toujours la disponibilité courante basée sur les données partenaires en temps réel et les fenêtres de droits.

## Tâches de maintenance

### Réindexation

Une réindexation complète doit être effectuée :

- Après des mises à jour massives de contenu
- Après des modifications de configuration des champs
- Après des mises à jour du module Search API
- Lorsque les mappings Elasticsearch changent

### Surveillance de la santé de l'index

La surveillance régulière doit vérifier :

- Taille de l'index et nombre de documents
- Métriques de performance des requêtes
- Opérations d'indexation en échec dans les logs
- Santé du cluster Elasticsearch
